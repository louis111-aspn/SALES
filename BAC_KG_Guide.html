SAP Knowledge Graph란 무엇인가

기술적 실체를 파악하고, 전통적 데이터 모델과의 구조적 차이를 ASPN Group 시나리오로 비교합니다.

1.1 기술적 실체 — 전통 데이터 모델과 무엇이 다른가

Knowledge Graph(KG)는 엔티티(Node) + 관계(Edge) + 속성(Property)로 구성된 그래프 데이터베이스 위에, 의미 체계(Ontology)를 얹은 데이터 아키텍처입니다. SAP가 "Knowledge Graph"를 선언할 때의 실체는 다음 세 가지 레이어입니다.

② Ontology — 비즈니스 개념의 공식 정의

온톨로지는 Triple에서 쓰이는 클래스(Node 종류)와 관계(Edge 이름)를 공식적으로 정의한 설계도입니다. ASPN Group 온톨로지 일부를 Turtle 형식으로 표현하면:

Turtle 형식 (.ttl) — ASPN Group 온톨로지 정의
# 네임스페이스 선언
@prefix aspn: <http://aspn.com/ontology#> .
@prefix rdf:  <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix owl:  <http://www.w3.org/2002/07/owl#> .

# 클래스(Node 종류) 정의
aspn:LegalEntity   rdf:type owl:Class .
aspn:Subsidiary    rdf:type owl:Class ;
                   rdfs:subClassOf aspn:LegalEntity .   # 상속 관계
aspn:Customer      rdf:type owl:Class .
aspn:SalesOrder    rdf:type owl:Class .
aspn:CostCenter    rdf:type owl:Class .

# 관계(Edge) 정의
aspn:subsidiaryOf  rdf:type owl:ObjectProperty ;
                   rdfs:domain aspn:Subsidiary ;
                   rdfs:range  aspn:LegalEntity .

aspn:placedBy      rdf:type owl:ObjectProperty ;
                   rdfs:domain aspn:SalesOrder ;
                   rdfs:range  aspn:Customer .

aspn:ownershipPct  rdf:type owl:DatatypeProperty ;
                   rdfs:range  xsd:decimal .

# 실제 데이터 인스턴스
aspn:ASPN_Korea    rdf:type         aspn:Subsidiary ;
                   aspn:subsidiaryOf aspn:ASPN_Holdings ;
                   aspn:ownershipPct "0.80"^^xsd:decimal ;
                   aspn:currency     "KRW" .

aspn:ASPN_Europe   rdf:type         aspn:Subsidiary ;
                   aspn:subsidiaryOf aspn:ASPN_Holdings ;
                   aspn:ownershipPct "1.00"^^xsd:decimal ;
                   aspn:currency     "EUR" .

③ SPARQL — 그래프 패턴 매칭 쿼리

동일한 비즈니스 질문 "ASPN Korea 담당 컨설턴트가 수주한 고객사 목록"을 SQL / MDX / SPARQL로 비교합니다.

SQL — 테이블 구조와 FK 관계를 알아야 함
SELECT DISTINCT c.customer_name
FROM   vbak o
JOIN   kna1  c  ON o.kunnr   = c.kunnr
JOIN   pa0001 sr ON o.pernr   = sr.pernr
JOIN   t_team t  ON sr.team_id = t.team_id
WHERE  t.team_name = 'BAC사업부'
  AND  o.bukrs     = 'ASPN_KR'
  AND  o.gbstk     = 'A';   -- 테이블명, 컬럼명, FK 구조 선행 지식 필요
MDX — 큐브 계층 구조를 알아야 함
SELECT
  [Customer].[Name].Members       ON COLUMNS,
  [Measures].[Order Count]        ON ROWS
FROM   [Sales_Cube]
WHERE  (
  [Team].[BAC사업부],
  [Company].[ASPN_Korea],
  [Order Status].[Open]
);   -- 큐브명, 차원명, 멤버명 선행 지식 필요
SPARQL — 관계 이름(Edge label)만으로 탐색 가능
SELECT DISTINCT ?customerName
WHERE {
  # 패턴 선언 — 테이블 구조 몰라도 됨
  ?rep       aspn:memberOf      aspn:BAC사업부 .
  ?rep       aspn:worksFor      aspn:ASPN_Korea .
  ?order     aspn:assignedTo    ?rep .
  ?order     aspn:placedBy      ?customer .
  ?order     aspn:status        "Open" .
  ?customer  aspn:name          ?customerName .
}
ORDER BY ?customerName
항목SQLMDXSPARQL
사고 방식테이블에서 꺼낸다축 교차로 집계한다패턴을 선언하면 매칭된다
사전 지식스키마/FK 필수큐브 계층 필수Edge 이름만으로 가능
집계 표현GROUP BY자연스러움 (강점)어색함 (약점)
학습 허들중높음중~높음 (사고 전환)
MDX 경험자 전환——2~3주면 문법 습득 가능
실무 포인트
SPARQL 문법보다 진짜 허들은 온톨로지 설계입니다 — Edge 이름을 누가, 어떻게 정의하는가. 이것이 SAP GR/CO 도메인 전문가가 KG 프로젝트에서 대체 불가능한 이유입니다.

시장성과 ROI 현실 진단

수요 지형, 고객사 ROI 구조, SAP 신제품 도입 사이클 패턴, KG가 실질적으로 해결하는 것과 남는 과제를 정리합니다.

3.1 수요 지형 — 대기업 vs 중견기업
실질 수요 있는 곳
  • 매출 5조+, 계열사 10개+ 대기업
  • 마스터데이터가 계열사별로 파편화된 곳
  • AI 도입 의지 + 예산이 동시에 존재
  • 데이터 통합 숙제가 이미 수년째 미해결
  • MDG나 Datasphere가 이미 도입된 곳
실질 수요 없는 곳
  • 중견기업 이하 — DW도 미완성 상태
  • 단순 보고/분석 목적 — BW/SAC으로 충분
  • 데이터 품질이 낮은 곳 — MDG가 먼저
  • AI를 실험 수준으로만 도입한 곳
  • IT 거버넌스가 약한 조직
시장성 판정
좁고 깊은 시장 — 대형 딜 소수. 국내에서 KG 도입 조건을 동시에 만족하는 기업은 현시점에서 손에 꼽힙니다.

공통 메커니즘

SAP는 구현 환경이 완비되기 전에 비전을 발표하고 라이선스를 선판매합니다. 이는 Enterprise SW 업계의 일반적 관행이기도 하나, SAP는 ERP 코어와의 통합을 "표준"으로 포장하기 때문에 고객 기대치가 높아지는 구조입니다.
SAP는 비전과 인프라를 제공하고, 실제 구현은 파트너(SI, 컨설팅사)가 담당합니다. 구현 실패 시 고객 불만은 파트너에게 향하는 구조입니다. Early Adopter Program은 파트너가 자비로 SAP 신제품을 검증해 주는 형태입니다.
Gartner, Forrester 등 분석 보고서에서 "리더" 포지션을 확보하면, CIO는 보고서를 근거로 도입을 결정합니다. 실제 구현 가능성은 보고서에 포함되지 않습니다.
